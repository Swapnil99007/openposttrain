# SFT Dataset Format

## Why chat-message JSONL

Every SFT/DPO trainer we're likely to use (TRL's `SFTTrainer`, `DPOTrainer`) expects
conversational data as a list of `{"role", "content"}` messages, not raw text. Writing
data in this format now means Stage 16 (LoRA SFT) can consume it with no reshaping.

## Record schema

One JSON object per line:

```json
{
  "messages": [
    {
      "role": "user",
      "content": "Solve the following math problem step by step.\n\nAt the end, write the final answer in this exact format:\n\nFinal Answer: <number>\n\nProblem:\n..."
    },
    {
      "role": "assistant",
      "content": "Reasoning...\n\nFinal Answer: 18"
    }
  ],
  "metadata": {
    "source": "gsm8k",
    "split": "train",
    "final_answer": "18"
  }
}
```

- `messages[0]` (user) is built from `prompts/gsm8k_v1.txt` via `format_prompt`, so
  the SFT model trains on the exact same prompt shape used at eval time.
- `messages[1]` (assistant) is GSM8K's gold solution with calculator annotations
  (`<<16-3=13>>13`) stripped, followed by a `Final Answer: <number>` line. This
  line is deliberately in the same format `extract_model_answer` in
  `src/openposttrain/evals/gsm8k.py` parses — the model is trained to produce
  exactly what the evaluator expects, closing the loop between eval and training.
- `metadata.final_answer` is the normalized gold answer (via `normalize_number`),
  useful for spot-checking or filtering without re-parsing `messages`.

## Split hygiene

`configs/data_gsm8k_sft_small.yaml` draws both `train` and `validation` records
from GSM8K's `train` split, at disjoint row ranges (validation starts right after
where train ends). GSM8K's `test` split is never read during data prep — it stays
fully held out for base-vs-SFT accuracy comparison (Stage 18), so SFT hyperparameters
are never tuned against the same examples used to measure final improvement.

## Current scope

`scripts/prepare_gsm8k_sft_data.py` generates the general-purpose GSM8K SFT set only.
The targeted failure-mode examples (age-direction errors, break-even vs.
starts-earning boundary conditions, per-serving vs. per-bag interpretation, etc. —
see the Qwen 512-token baseline failure analysis in `PROJECT_LOG.md`) are a separate,
hand-curated batch to be added on top of this file later, not generated by this script.
